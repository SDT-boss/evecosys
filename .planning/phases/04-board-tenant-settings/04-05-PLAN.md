<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → PATCH /api/board/settings/toggles | Untrusted flags payload — injection risk via unknown keys |
| JSONB column | Full replacement on update — partial payload would silently wipe flags |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05-01 | Tampering | PATCH /api/board/settings/toggles — flag key injection | mitigate | Route validates that `Object.keys(flags)` is a strict subset of KNOWN_FLAGS (8 keys) AND all 8 required keys are present — unknown keys rejected with 400; payload cannot introduce arbitrary JSONB properties |
| T-04-05-02 | Tampering | PATCH /api/board/settings/toggles — value type injection | mitigate | Route validates `typeof v === 'boolean'` for every value — string 'yes'/'no', null, numbers all rejected with 400 |
| T-04-05-03 | Tampering | JSONB partial update — flag wipe | mitigate | Client always sends all 8 flags; server requires all 8 keys present (validation fails on incomplete payload); full object replacement in UPDATE (not jsonb_set) is intentional and safe |
| T-04-05-04 | Elevation of Privilege | PATCH /api/board/settings/toggles — cross-tenant write | mitigate | Tenant fetched via `owner_id = auth.uid()` — board member can only update their own tenant's feature_flags; RLS policy `tenants_update_own` provides a second layer |
| T-04-05-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan |
</threat_model>
