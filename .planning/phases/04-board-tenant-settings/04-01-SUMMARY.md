# Phase 4 Plan 01: DB Migrations and Wave 0 Test Scaffolding Summary

**One-liner:** Three SQL migrations add branding columns, feature_flags JSONB, and users.tenant_id FK with RLS policies and updated handle_new_user trigger; eleven stub files scaffold the test harness before any Wave 1 routes exist.

## What Was Built

### Task 1: DB Migrations

Three migration files created and applied to local Supabase:

**`20260620120000_add_tenant_branding.sql`**
- Adds `logo_url TEXT` and `primary_color TEXT` to `public.tenants` (nullable, no DEFAULT)
- Creates `tenant-assets` storage bucket (public) via `INSERT INTO storage.buckets ON CONFLICT DO NOTHING`
- Creates `tenant_owner_upload` RLS policy on `storage.objects` using `split_part(name, '/', 2)` to scope uploads to the tenant owner's UUID path

**`20260620130000_add_tenant_feature_flags.sql`**
- Adds `feature_flags JSONB NOT NULL DEFAULT '{"member_invitations":true,...}'::jsonb` to `public.tenants`
- All 8 flags (member_invitations, fleet, carbon, trips, driver_behaviour_score, alerts, charging_stations, auth_troubleshooting) default to `true`

**`20260620140000_add_users_tenant_id.sql`**
- Adds `tenant_id UUID REFERENCES public.tenants(id) ON DELETE SET NULL` to `public.users`
- Creates `users_select_by_tenant` RLS policy: board members see only users in their tenant
- Creates `users_delete_board_own_tenant` RLS policy: board members can hard-delete users in their tenant
- Rewrites `handle_new_user()` via `CREATE OR REPLACE FUNCTION`: preserves all existing logic (provider_id variable, email-existence check, UPDATE public.drivers, UPDATE public.users) and adds `tenant_id = COALESCE((NEW.raw_user_meta_data->>'tenant_id')::uuid, tenant_id)` to the UPDATE and `tenant_id` to the INSERT column list

## Threat Surface Scan

The storage RLS policy `tenant_owner_upload` guards the trust boundary at `storage.objects`. The path-segment check (`split_part(name, '/', 2) = tenant_id`) ensures uploads are scoped to the authenticated user's own tenant UUID. This was explicitly in the plan's threat register (T-04-W0-01 — disposition: mitigate, applied).

No new trust surface beyond what is in the plan's threat model was introduced.
