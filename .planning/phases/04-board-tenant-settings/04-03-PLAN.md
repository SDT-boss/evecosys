<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/board/settings/users/invite | Caller supplies email + role; role injection risk |
| client → DELETE /api/board/settings/users/remove | Caller supplies userId; must not delete users from other tenants |
| Service-role client | Has full auth admin access — used only inside API routes, never client-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Elevation of Privilege | POST /api/board/settings/users/invite — role injection | mitigate | API route validates `role` against `['manager', 'driver']` allowlist — `platform_admin` and `board` rejected with 400; client-side also validates but server is authoritative |
| T-04-03-02 | Elevation of Privilege | DELETE /api/board/settings/users/remove — cross-tenant delete | mitigate | Delete query uses `.eq('tenant_id', tenant.id)` where tenant.id comes from `owner_id = auth.uid()` lookup — attacker cannot delete users from another tenant even with a valid userId |
| T-04-03-03 | Information Disclosure | Service-role client (`createServiceClient`) | mitigate | `createServiceClient()` called only inside API route handler (never at module scope, never in Client Components) — service key stays server-side |
| T-04-03-04 | Tampering | inviteUserByEmail metadata — tenant_id forgery | mitigate | `tenant_id` in invite metadata comes from the server-side tenant lookup (`owner_id = auth.uid()`) — client cannot supply a different tenant_id to hijack invite targeting |
| T-04-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan |
</threat_model>
