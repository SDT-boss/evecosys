<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Migration execution | SQL executed against live Supabase; malformed SQL can corrupt schema |
| Storage RLS | If path-prefix check in RLS is wrong, any authenticated user could write to any tenant's storage path |
| Trigger function | `handle_new_user` runs as SECURITY DEFINER — must not leak tenant_id from other tenants |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-W0-01 | Tampering | storage.objects RLS policy tenant_owner_upload | mitigate | Use `split_part(name, '/', 2) = (SELECT id::text FROM public.tenants WHERE owner_id = auth.uid())` to scope uploads to the authenticated user's own tenant path; the tenant UUID in the path cannot be forged without owning the tenants row |
| T-04-W0-02 | Elevation of Privilege | handle_new_user trigger — tenant_id write | mitigate | The trigger reads `raw_user_meta_data->>'tenant_id'` cast to UUID; an invalid UUID string fails the cast gracefully (NULL result); the trigger does not allow arbitrary tenant assignment — only the invite call (service-role client, controlled by board member's API route) sets this metadata |
| T-04-W0-03 | Information Disclosure | users_select_by_tenant RLS policy | mitigate | Policy uses subquery `(SELECT id FROM public.tenants WHERE owner_id = auth.uid())` which returns NULL for non-board users, causing the policy to match zero rows — board members see only their own tenant's users |
| T-04-W0-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan — all dependencies already present |
</threat_model>
