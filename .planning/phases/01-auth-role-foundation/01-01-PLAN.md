<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Next.js layout | Unauthenticated or wrong-role user requests hit layout server components; auth check enforced here |
| Next.js server → Supabase DB | Server helper calls set_config before queries; all DB access uses anon key + RLS, never service key |
| DB session variable → RLS policy | `app.active_tenant_id` set per transaction; RLS reads it to scope platform admin queries |
| seed.sql → local Supabase | Hardcoded dev credentials; must never reach production |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Elevation of Privilege | users.role CHECK constraint | mitigate | Only the server-side trigger (SECURITY DEFINER) inserts into public.users.role; users cannot call UPDATE on their own role row — existing RLS users_update_own policy does not allow role column mutation |
| T-01-02 | Tampering | raw_user_meta_data role claim via Supabase auth API | mitigate | Even if a user sets their own raw_user_meta_data.role = 'platform_admin', the CHECK constraint on public.users.role is the authoritative gate; the trigger reads meta_data but the DB rejects invalid values; route guards read from public.users.role, not auth.users metadata |
| T-01-03 | Information Disclosure | set_config / connection pooling tenant bleed | mitigate | set_config called with third arg true (transaction-local) — variable resets at transaction end; in Supabase's PgBouncer transaction-mode pooling this is the safe and standard pattern; no cross-request bleed |
| T-01-04 | Information Disclosure | tenants RLS — missing platform_admin SELECT policy | mitigate | tenants_select_platform_admin policy added in this plan's migration; without it platform admin queries return zero rows (silent data starvation, not exposure) |
| T-01-05 | Information Disclosure | seed.sql hardcoded credentials in repo | accept | Credentials (DevPassword123!) are local-dev-only; seed.sql is committed alongside migrations as standard Supabase practice; production uses environment variables and has no seed.sql applied; risk is developer credential exposure in repo — LOW for a private repo |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this phase; all dependencies already in package.json; no legitimacy checkpoint needed |
</threat_model>
