# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Auth & Role Foundation.
> Generated by gsd-ui-researcher. This is a **retroactive contract** — the stub pages already exist
> and have been UAT-verified. This spec documents the design decisions governing those stubs and
> locks the token/spacing/color baseline so Phases 2, 4, and 5 stay consistent.

---

## Context Note

Phase 1 is a pure-infrastructure phase. The only UI deliverables are two intentionally-minimal
stub pages that exist solely to make AUTH-02 and AUTH-03 testable:

- `app/(dashboard)/platform/page.tsx` — stub for `/platform` (Phase 2 builds the real shell)
- `app/(dashboard)/board/settings/page.tsx` — stub for `/board/settings` (Phase 4 builds the tabs)

Neither stub uses `DashboardShell` or any component library. Both use inline `style` props with
`var(--ds-*)` custom properties from `design-system/tokens/variables.css` and semantic
`var(--text*)` aliases defined in `app/globals.css`.

**Token discrepancy (documented):** Both stub pages currently use `var(--ds-spacing-6)`. This
token does not exist in `design-system/tokens/variables.css`. The design token for 24px is
`var(--ds-space-lg)`. The stubs resolve `--ds-spacing-6` to `undefined` (falls through to 0).
Phase 2 MUST correct this to `var(--ds-space-lg)` when building the real platform shell. This
discrepancy does not affect the test outcomes for AUTH-02 or AUTH-03 (tests check redirect
behavior, not visual layout). The correct token is locked below.

## Spacing Scale

Declared values from `design-system/tokens/variables.css` (all multiples of 4):

| Token | CSS Custom Property | Value | Usage in Phase 1 |
|-------|--------------------|----|-------------------|
| xs | `var(--ds-space-xs)` | 4px | Icon gaps, inline padding (not used in stubs) |
| sm | `var(--ds-space-sm)` | 8px | Compact element spacing (not used in stubs) |
| md | `var(--ds-space-md)` | 16px | Default element spacing (not used in stubs) |
| lg | `var(--ds-space-lg)` | 24px | **`<main>` padding on both stub pages** (intended; stubs use non-existent `--ds-spacing-6` — see discrepancy note above) |
| xl | `var(--ds-space-xl)` | 32px | Layout gaps (not used in stubs) |
| 2xl | `var(--ds-space-2xl)` | 48px | Major section breaks (not used in stubs) |
| 3xl | `var(--ds-space-3xl)` | 64px | Page-level spacing (not used in stubs) |

Exceptions: none. The 8-point scale applies throughout. Touch-target exceptions (44px) are not
relevant for Phase 1 (no interactive elements in stubs).

**Canonical padding for stub `<main>`:** `padding: var(--ds-space-lg)` (24px).

Declared from `design-system/tokens/variables.css` and `app/globals.css`:

| Role | Token | Computed Value | Weight | Line Height | Phase 1 Usage |
|------|-------|----------------|--------|-------------|---------------|
| Body base | `--ds-font-size-base` | 16px (1rem) | 400 (`--ds-font-weight-normal`) | 1.5 (`--ds-font-line-height-normal`) | Global default via `globals.css` body (14px Barlow) |

decide whether the platform shell switches to `--ds-font-family-sans` or inherits Barlow.

## Color

All values from `design-system/tokens/variables.css` (ds-* tokens) and `app/globals.css` (semantic
aliases). Light mode values shown; dark mode via `.dark` class in globals.css.

| Role | CSS Custom Property | Light Mode Value | Dark Mode Value | Usage |
|------|--------------------|--------------------|-----------------|-------|
| Dominant (60%) — page background | `var(--bg)` | `#f2f5f2` | `#111412` | Page background (body-level) |
| Secondary (30%) — surfaces | `var(--surface)` | `#ffffff` | `#1c201c` | Card / panel surfaces |
| Secondary (30%) — surface alt | `var(--surface2)` | `#f7f9f7` | `#242824` | Raised surfaces, sidebar fills |
| Text — primary | `var(--text)` | `#1a1a1a` | `#e8f0e8` | `<h1>` in both stubs (heading color) |
| Text — tertiary | `var(--text3)` | `#8a9a8a` | `#6a7a6a` | `<p>` placeholder body in both stubs |
| Brand accent | `var(--ds-color-brand-primary)` | `#008684` | — | Navigation, links, focus rings (NOT in Phase 1 stubs) |
| CTA accent | `var(--ds-color-brand-secondary)` | `#96d02c` | — | Primary CTA, active states (NOT in Phase 1 stubs) |
| Destructive | `var(--ds-color-status-error)` | `#ef4444` | — | Destructive actions only (NOT in Phase 1) |

**60/30/10 split (Phase 1 stubs):**

- 60% dominant: `var(--bg)` — the page background color inherited by `<main>`
- 30% secondary: `var(--surface)` / `var(--surface2)` — not rendered by stubs directly; inherited from parent shell wrappers in Phase 2
- 10% accent: `var(--ds-color-brand-primary)` (Cyber Jade) — reserved for navigation, links, focus rings. `var(--ds-color-brand-secondary)` (Volt Green) reserved for primary CTA. Neither appears in Phase 1 stubs; locked here so Phase 2 uses them correctly.

Accent reserved for:
1. Navigation active/hover states — `var(--ds-color-brand-primary)` (#008684 Cyber Jade)
2. Primary CTA button fill — `var(--ds-color-brand-secondary)` (#96d02c Volt Green)
3. Focus rings — `var(--ds-color-brand-primary)` (#008684 Cyber Jade)
4. Link text — `var(--ds-color-brand-primary)` (#008684 Cyber Jade)

## Interaction Contract

Phase 1 stubs have no interactive elements. All routing behavior is server-side.

| Behavior | Mechanism | Where |
|----------|-----------|-------|
| `/platform` access denied → `/login` | `redirect('/login')` in `platform/layout.tsx` | Server component, no client JS |
| `/board/settings` access denied → `/login` | `redirect('/login')` in `board/settings/layout.tsx` | Server component, no client JS |
| No tenant row for board user → `/login` | `if (!tenant) redirect('/login')` | Secondary guard in `board/settings/layout.tsx` |

No loading states, hover states, focus styles, or transitions exist in Phase 1 stubs.
Phase 2 introduces interactive elements and must define their states in `02-UI-SPEC.md`.

## Layout Pattern

Both stub pages use the same minimal layout:

```
<main style={{ padding: 'var(--ds-space-lg)' }}>
  <h1 style={{ color: 'var(--text)' }}>{heading}</h1>
  <p style={{ color: 'var(--text3)' }}>{placeholder}</p>
</main>
```

**No `DashboardShell` wrapper** — Phase 1 layouts return `<>{children}</>` directly (no nav, no
topbar, no sidebar). The authenticated user sees a bare page with heading + placeholder text.
Phase 2 must wrap `/platform` in its own shell. Phase 4 must nest `/board/settings` within the
existing `DashboardShell` provided by `board/layout.tsx`.
