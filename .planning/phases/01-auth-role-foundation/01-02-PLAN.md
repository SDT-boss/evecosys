From app/(dashboard)/manager/layout.tsx — canonical four-step role guard skeleton (copy-exact, two substitutions):
```typescript
import { redirect } from 'next/navigation'
import { createClient } from '@/lib/supabase/server'
// Note: DashboardShell / AlertBellWrapper / NAV are NOT imported for platform layout
// (Phase 2 builds the platform shell; Phase 1 stub uses minimal HTML)

export default async function PlatformLayout({ children }: { children: React.ReactNode }) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  if (!user) redirect('/login')

  const { data: profile } = await supabase
    .from('users')
    .select('*')
    .eq('id', user.id)
    .single()

  if (!profile || profile.role !== 'platform_admin') redirect('/login')

  return <>{children}</>
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request → Next.js layout server component | All role enforcement happens in layout server components; no middleware; browser cannot bypass server-side redirect |
| layout.tsx → Supabase DB (users table) | Profile fetch uses anon key + RLS users_select_own policy; no elevation possible |
| layout.tsx → Supabase DB (tenants table) | Tenant ownership fetch uses anon key + tenants_select_own policy; board user can only read their own row |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | platform/layout.tsx role check | mitigate | Guard checks profile.role from public.users (server-fetched, RLS-protected); not from client-side token claim or cookie value; attacker cannot forge a platform_admin role without a valid DB row |
| T-02-02 | Elevation of Privilege | board/settings/layout.tsx tenant bypass | mitigate | Two independent checks: (1) role must be 'board', (2) a tenants row WHERE owner_id = auth.uid() must exist; both must pass; nesting inside board/layout.tsx adds a third check (parent guard) as defense-in-depth |
| T-02-03 | Information Disclosure | board/settings stub reveals route existence to unauthorized users | accept | Route redirects to /login; no data or route structure is exposed; stub content only renders for authorized users; acceptable for this threat level |
| T-02-04 | Tampering | layout.tsx bypassed if middleware intercepts first | accept | No middleware.ts exists in this project (verified in RESEARCH.md); all guards are in layout.tsx files; risk is near-zero; any future middleware addition must re-examine this |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan; all dependencies already in package.json |
</threat_model>
