## Interaction Contract

### State Machine: TenantList Row Click

The tenant-switching flow has four states. State transitions are driven by `useTransition` (`isPending`) and local React state (`pendingTenantId`, `activeTenantId`, `switchError`).

```
IDLE ──[row click]──► OPTIMISTIC + LOADING ──[action ok]──► IDLE (new active tenant)
                                           └──[action fail]──► ERROR (previous active tenant restored)
```

**State: IDLE**
- All rows interactive: `pointer-events: auto`, `opacity: 1`, `cursor: pointer`
- Active row: `data-state="selected"` → `bg-[var(--ds-color-neutral-grey-10)]`
- No Spinner visible anywhere in the list
- No Alert visible

**State: OPTIMISTIC + LOADING (isPending = true)**
- Local `activeTenantId` immediately set to the clicked tenant ID (optimistic)
- The newly-clicked row shows: `data-state="selected"` highlight + `<Spinner size="sm">` replacing the tenant name text
- All rows: `<TableBody>` gets `opacity: 0.5`; `<Table>` wrapper gets `cursor: not-allowed`; each `<TableRow>` gets `pointer-events: none` (prevents click-through)
- No Alert visible
- `ActiveTenantIndicator` in header reflects the optimistic tenant name immediately (requires TenantList to accept an `onOptimisticChange` callback or share state with the layout — see Integration note below)
- `aria-busy="true"` on the `<Table>` element for screen readers

**State: IDLE (success)**
- `isPending` resolves to `false`
- `pendingTenantId` cleared to `null`
- `activeTenantId` local state already holds the new tenant ID (optimistic was correct)
- `revalidatePath('/platform', 'layout')` fires from the server action — layout re-renders, `ActiveTenantIndicator` confirms via server-rendered name
- No additional success UI (D-04: header update is the success confirmation)
- `aria-busy="false"` on the `<Table>` element

**State: ERROR (action returned ok: false)**
- Local `activeTenantId` reverted to the previous value
- `pendingTenantId` cleared to `null`
- `switchError` set to the error message string
- All rows return to fully interactive (opacity 1, pointer-events auto)
- `<Alert variant="destructive">` renders above the `<Table>` with the error message
- Alert is manually dismissible via an X button (see Alert dismissal below)

### Spinner Placement in TableRow

The `<Spinner size="sm">` (16×16px) replaces the tenant name text within `<TableCell>` on the loading row — it does not sit alongside the name. Implementation:

```tsx
<TableCell>
  {pendingTenantId === tenant.id ? (
    <span className="flex items-center gap-[var(--ds-space-xs)]">
      <Spinner size="sm" />
      <span className="sr-only">Switching to {tenant.name}…</span>
    </span>
  ) : (
    tenant.name
  )}
</TableCell>
```

Rationale (Claude's discretion): Replacing the name with the Spinner (not showing both alongside) keeps the cell width stable and avoids horizontal layout shift. The `sr-only` span communicates the in-progress context to screen readers. The tenant name is still visible on the status badge cell in the same row.

### Alert Dismissal Mechanism

The error Alert is manually dismissible only (no auto-dismiss). Implementation: render an `×` button (`aria-label="Dismiss error"`) inside the Alert that calls `setSwitchError(null)`. No timeout.

**Rationale (Claude's discretion):** Auto-dismiss (e.g. 5 seconds) risks the admin not reading the error message before it disappears, leaving them confused about why the active tenant did not change. Manual dismissal ensures the error is acknowledged. The Alert disappears naturally if the admin successfully switches to a different tenant (the component re-renders without an error).

Alert dismissal renders as:

```tsx
<Alert variant="destructive" className="mb-[var(--ds-space-sm)]">
  <AlertTitle>Switch failed</AlertTitle>
  <AlertDescription>{switchError}</AlertDescription>
  <button
    aria-label="Dismiss error"
    className="absolute right-[var(--ds-space-md)] top-[var(--ds-space-md)] text-[var(--ds-color-status-error)] hover:opacity-70"
    onClick={() => setSwitchError(null)}
  >
    ×
  </button>
</Alert>
```

### Blocked Screen — platform/layout.tsx Guard

**Trigger condition:** Platform admin navigates to any `/platform` sub-route where `requiresActiveTenant` is true AND the `platform_active_tenant` cookie is absent or empty.

**Sub-route boundary (Claude's discretion):** The tenant list page (`/platform`) does NOT require an active tenant — it is the selection surface. All routes nested beyond `/platform` (e.g. `/platform/tenants/[id]`, `/platform/reports`, any future sub-routes) require an active tenant. The layout detects this by comparing `pathname` against `/platform` root: if `pathname !== '/platform'` AND cookie is absent → render `<BlockedScreen>`.

Implementation in `layout.tsx`:

```tsx
import { headers } from 'next/headers'

// After cookie read:
const headerStore = await headers()
const pathname = headerStore.get('x-pathname') ?? ''
const isSubRoute = pathname !== '/platform' && pathname !== '/platform/'

if (isSubRoute && !tenantId) {
  return (
    <PlatformShell user={profile as AppUser} activeTenantName={null}>
      <BlockedScreen />
    </PlatformShell>
  )
}
```

> Note: `x-pathname` must be forwarded by Next.js middleware, or `usePathname()` used in a client wrapper. The planner must choose the appropriate Next.js 16 approach for reading the current pathname in a Server Component layout.

**BlockedScreen renders within PlatformShell** — the topbar/nav chrome remains visible so the admin always has navigation access. Only `{children}` is replaced.

## Integration Notes

### ActiveTenantIndicator Optimistic Update

The `ActiveTenantIndicator` lives in `PlatformShell`, which is a client component receiving `activeTenantName` as a prop from the Server Component layout. For Phase 3 optimistic UI to update the header immediately on row click (before the server responds), two approaches are possible:

**Approach A (recommended):** Lift `activeTenantId` and `activeTenantName` state into a React Context provided by `PlatformShell`. `TenantList` reads from and writes to this context. When a row is clicked, the context is updated immediately with the new tenant name (the name is already in `TenantList`'s props). On layout re-render (after `revalidatePath`), the server-provided prop re-seeds the context.

**Approach B (simpler, deferred complexity):** `TenantList` accepts an `onActiveTenantChange(name: string)` callback prop. The layout passes this down through `PlatformShell` → `TenantList`. On row click, the callback fires immediately with the new tenant name; PlatformShell updates its local `activeTenantName` state.

The planner selects the implementation approach. Either way, `ActiveTenantIndicator` must receive the optimistic name update within the same synchronous event handler as the `startTransition` call — no additional server round-trip for the success signal.

### setActiveTenant Return Type Change

The existing `setActiveTenant` at `app/(dashboard)/platform/actions.ts` returns `Promise<void>`. Phase 3 changes this to `Promise<{ ok: boolean; error?: string }>`. The `TenantList` component currently calls it without awaiting the result inside a transition. Phase 3 must:

1. Update the server action signature.
2. Await the result inside `startTransition` to check `result.ok`.
3. On `!result.ok`: revert optimistic state + set `switchError`.

## Decisions Made (Claude's Discretion)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Disabled-row visual treatment | Opacity `0.5` on `<TableBody>` + `cursor: not-allowed` on `<Table>` wrapper + `pointer-events: none` on each `<TableRow>` | Both opacity and cursor signal non-interactivity; opacity on TableBody only (not header) keeps column labels legible; per-row `pointer-events: none` prevents click-through without requiring JS guards |
| Spinner placement within TableRow | Replaces tenant name text in `<TableCell>` (not alongside) | Prevents horizontal layout shift; cell width stays stable; tenant name still readable via adjacent status cell |
| Alert dismissal | Manual only — X button sets `switchError(null)` | Auto-dismiss risks the admin missing the error message; manual ensures acknowledgement |
| BlockedScreen as own component | Extracted to `components/platform/BlockedScreen.tsx` | Required for Phase 5 Storybook coverage; separation of concerns in layout file |
| Blocked screen sub-route boundary | Any pathname !== `/platform` AND !== `/platform/` | `/platform` (tenant list) is the selection surface and must always be reachable; all nested routes require active tenant |
| EmptyState title (BlockedScreen) | `"Select a workspace to continue"` | Imperative, action-oriented; tells admin exactly what to do |
| EmptyState description (BlockedScreen) | `"This area requires an active workspace. Choose one from the tenant list to get started."` | Explains the "why" without cookie/technical jargon; maps to admin mental model |
| BlockedScreen CTA label | `"Go to tenant list"` | Specific destination; avoids vague labels like "Go back" |
| Alert title | `"Switch failed"` | Short, scan-friendly; pairs with description for full context |
| Error description format | Base sentence + optional parenthetical for server error | Keeps base message non-technical; appends technical detail when available for debugging |
