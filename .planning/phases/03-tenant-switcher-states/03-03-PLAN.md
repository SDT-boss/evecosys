<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser request → middleware → layout RSC | URL path passes through middleware (trusted) which sets x-pathname; layout reads and uses it only for rendering decisions — no user data involved |
| Cookie absence → BlockedScreen render | Layout reads cookie server-side; absence of cookie is a safe-to-trust signal for rendering BlockedScreen (cannot be spoofed by client to bypass the guard) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-W2-01 | Elevation of Privilege | platform/layout.tsx blocked-screen bypass | mitigate | Guard runs in RSC layout on every server render; client-side navigation still triggers a server request that executes the layout guard; cannot be bypassed by manipulating client-side state. Confirmed: layout RSC reads cookie server-side per request. |
| T-03-W2-02 | Spoofing | x-pathname header injection | accept | Header is set by Next.js middleware running server-side; clients cannot inject arbitrary headers that pass through the middleware layer in Next.js App Router. Low risk: the header is used only for a rendering branch decision, not for auth or data access. |
| T-03-W2-03 | Tampering | E2E test cookie clearing leaving stale auth | accept | E2E clearCookies() removes the tenant cookie but Supabase session is in localStorage; tests clean up storage state per test.use() pattern; no cross-test contamination. |
| T-03-W2-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed; slopcheck not required. |
</threat_model>
