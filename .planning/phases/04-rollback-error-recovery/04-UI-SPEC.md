## Copywriting Contract

Phase 4 introduces one domain error class (`ProvisioningRollbackError`) whose message string is directly user-facing via API responses and future UI surfaces. All other error messages in Phase 4 are existing and unchanged.

### Error Message Strings (Phase 4 deliverables)

| Element | Copy |
|---------|------|
| `ProvisioningRollbackError` message template | `Provisioning rollback failed: {rollbackError.message} (original error: {originalError.message})` |
| `ProvisioningRollbackError` — `name` field | `"ProvisioningRollbackError"` |
| Pre-store failure (connectivity probe rejects) | Existing: `"Connectivity probe failed: {reason}"` — unchanged |
| Post-store transition failure only (clean rollback) | Existing: original error re-thrown — unchanged |

### Future UI Copy (for the frontend phase that surfaces these errors)

These are the copy contracts that the Phase 4 error surface implies. They are not implemented in Phase 4 but are locked here so the future frontend phase has a consistent contract.

| Element | Copy |
|---------|------|
| Provisioning failed, rollback succeeded heading | `"Provisioning failed"` |
| Provisioning failed, rollback succeeded body | `"Your database credentials could not be verified. Your account has been reset to the Registered state — you can try again when ready."` |
| Provisioning failed, rollback also failed heading | `"Critical provisioning error"` |
| Provisioning failed, rollback also failed body | `"Provisioning failed and automatic cleanup could not complete. Contact support with error ID {id} so we can manually clear your account."` |
| Tenant state pill — Registered (after rollback) | `"Registered"` |
| Tenant state pill — Provisioning (in progress) | `"Provisioning…"` |
| Retry CTA label | `"Retry Registration"` |
| Primary registration CTA label | `"Register Database"` |
| Empty state heading (no active tenants) | `"No tenants provisioned"` |
| Empty state body | `"Register your first database to get started."` |

### Destructive Actions in Phase 4

Phase 4 introduces no destructive user-initiated actions. Vault credential deletion during rollback is automatic and system-initiated — no user confirmation required or possible.

| Action | Triggered by | Confirmation |
|--------|-------------|--------------|
| `vault.delete(secretId)` during rollback | Automatic on post-store failure | None — system-initiated, no user interaction |
